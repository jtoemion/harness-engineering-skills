Persona Brainstorm — How-To Plan

Methodology Grill-with-Docs 2 Personas 3 ADRs per session Pastpapr LMS — 2026-05-30
This document is the full reference for running a Persona Brainstorm session — a structured client-dev discovery exercise that surfaces one high-impact concrete improvement per session. It covers the concept, role setup, step-by-step execution, the ADR format, and a complete roadmap for deploying the method on any project, illustrated with the Pastpapr LMS session from May 30, 2026.

1. What Is a Persona Brainstorm Session

A Persona Brainstorm session is a focused discovery exercise between two characters — a developer and a client — with a silent scribe documenting the outcome. The goal is not to enumerate features or vote on a backlog. It is to find one high-impact improvement that meaningfully changes the product, grounded in real contradictions, not aspirations.

It is called a "grill" session because the developer pushes — one question at a time — until the client gives a concrete pain statement, not a vague wish. The developer reads the codebase and project history first so every question is grounded in evidence, not assumptions.

Why it works

  • Developers ask better questions when they've read the docs first. They notice anomalies: features marked "done" but with untested next steps, systems marked "in_progress" for months, deprecated code that still deploys, contradictions between code and history.
  • Clients speak more precisely when they have full project context. Before the session, load Jeremiah (the client persona) with all project documentation. When a client answers from real knowledge — not guesses — the friction surfaces faster.
  • The ADR format forces specificity. "Improve the UX" is not an ADR. "Add a 3-field TutorSessionNoteModal with a Firestore write" is. If a developer can read it and start building without asking a follow-up question, it's a good ADR.
  • Three ADRs is the cap. Discovery without resolution is conversation. Three finished ADRs is a strong session. More than three means nothing was committed deeply enough.

2. The Three Personas

NameRoleResponsibility
Ezekiel Developer Reads all project docs (.docs/, .memory/). Asks one pointed question at a time. Translates client answers into architectural decisions. Writes ADRs in real time. Never monologues.
Jeremiah Client Loaded with full project context. Answers from real product knowledge, user pain, workflow gaps — not aspirations. Speaks in concrete examples. Not Judah. The user does not answer as Jeremiah. Jeremiah is the product owner.
Megumi Observer / Scribe Documents CLOSED questions to CONTEXT.md with G-Q# IDs. Writes completed ADRs to ADR.md. Silent unless format corrections are needed. Pushes output to correct repo immediately after the session.

Critical rule: Jeremiah is not the user

If Judah (the user) answers a question instead of Jeremiah, Megumi redirects: "Let's keep Jeremiah in character — what's the client answer here?". This prevents the session from collapsing into a free-form chat and forces every answer to come from product context.

Phase 1 — Orientation

Ezekiel opens. Not with "what do you want?" but with a specific, documented contradiction found from reading the code:

  • "The report card is marked LIVE but the next step in the progress log is a manual E2E test. Did that ever get verified?"
  • "League reset is manual only, but the Firebase Cloud Functions with the old XP thresholds are still deployed. Is this a dual-write risk?"
  • "EPIC-1 is listed 'in_progress' after months with no commits. Is the data still split across two curriculum models?"

Good opening questions share a structure: specific observation + implicit "which is the live state?". They come from the codebase, not from the user's stated preferences.

Phase 2 — Discovery (3 Rounds)

  • Round 1: One question about the product loop. How do users actually engage daily vs monthly? Closes surface-level questions fast.
  • Round 2: Pushes on the thread that seemed most real from Round 1. Ezekiel presses until Jeremiah gives a concrete pain statement — not "could we have X" but "we keep running into Y."
  • Round 3: Pivot to a second theme if one exists. If Jeremiah gives a soft answer, Ezekiel pushes: "Can you give me a specific example of when that came up?"

Phase 3 — ADR Writing

After each confirmed improvement:

  1. Ezekiel states the context — what the problem is, one sentence
  2. Jeremiah responds — what users are actually feeling
  3. Ezekiel writes the ADR using the template in real time
  4. Megumi confirms it was saved to ADR.md

Phase 4 — Skill Save

After the session: offer to save the method as a reusable skill with full session example. The skill lives in ~/.hermes/skills/persona-brainstorm/SKILL.md and gets pushed to jtoemion/harness-engineering-skills.

4. The ADR Format

ADR Template

Context What's wrong or missing. One paragraph — specific, not aspirational. No "wouldn't it be nice."
Decision What we're doing. Concrete and specific. Include specific files/components if applicable.
What to build (v1) Concrete scope — specific files, features, UI components. Be surgical to prevent scope creep.
What is deliberately excluded v1 boundaries. Explicitly stated so scope doesn't creep post-hoc.
Why this is high-impact Product-level rationale — how does this change the user's week, not the code's structure?
Verification Specific acceptance criteria a developer can run — real tests, not "should work well."

7. ADRs Generated — Pastpapr LMS

ADR-001 TutorSessionNoteModal — Session Observation Capture
Context The Pastpapr loop is quiz-driven. Students earn XP, move up league tiers, complete quizzes — that works. But for tutors, there's no capture mechanism for what happens during a 1-on-1 session. Tutors observe specific things — what a student understood, what tripped them up, what to follow up on — and that information lives nowhere in the system. The product is looking backward without giving tutors a way to record what they're seeing in real time.
Decision Add a lightweight structured session observation form linked to a student's profile. Three fields only: what the student understood, what the吾士 tripped them up, what to practice next. Output is a tutor note attached to the student document, visible at the next session and available as context when generating a report card.
What to build (v1) DAL: src/lib/dal/tutorNotesDAL.ts — create/read/list tutor notes by studentId
Service: tutorNoteService.ts — createTutorNote(studentId, {understood, struggled, practiceNext})
Hook: useTutorNotes.ts — list by studentId
UI: TutorSessionNoteModal.tsx — 3-field modal, inline in session flow
Firestore: tutorNotes/{noteId} with studentId, classId, tutorId, understood, struggled, practiceNext, createdAt
Wire from: AttendanceConfirmationDialog.tsx "Add note" button
Integration: LLM report card prompt reads recent tutor notes as Layer 4 context injection
What is deliberately excluded Free-text entry (v1: structured fields only) · Parent/student-facing note view · Automated task generation · Multi-student bulk note entry
Why this is high-impact Tutors currently manage this mentally or in a separate system. Structured note capture makes Pastpapr feel like a teaching tool rather than a quiz engine. Notes create continuity between sessions — a student's next visit shows what they were struggling with last week, not just an XP counter.
Verification Create a note from attendance confirmation → note appears in student profile · Note visible on next session in SessionSection · LLM narrative references recent tutor note (if present) · npm run test:run passes
ADR-002 Curriculum Model Excision — Remove Deprecated Paths
Context Tutors have fully migrated to ClassworkTopic → ClassworkSubtopic (new model). The old Topic → SubjectMatter model is no longer used in production Firestore. The migration script exists but was never executed because the problem solved itself naturally. What remains is dead code that misleads new developers about which path is canonical.
Decision Remove all code for the old curriculum model. Not a migration — full excision. No data migration required because the old collection is already empty.
What to build (v1) Delete: src/lib/dal/curriculumDAL.ts (entire file)
Remove: deprecated Topic/SubjectMatter methods from curriculumService.ts
Remove: Topic, SubjectMatter types from src/types/curriculum.ts
Remove: topics, subjectMatter match patterns from firestore.rules
Remove: old collection indexes from firestore.indexes.json
Keep: scripts/migrate-curriculum-to-classwork.js as historical reference, mark "DO NOT EXECUTE"
What is deliberately excluded Data migration script execution (not needed — old data is already empty) · New model DAL/service/UI changes (already correct)
Why this is high-impact New developers reading the codebase immediately see one canonical model. Removes accidental consumer risk. Cleaner architecture presentation for code reviews and onboarding. The codebase stops lying about what is real.
Verification npm run test:run — all pass, no old-model references in test mocks · npm run build clean · ESLint clean · Manual: create ClassworkTopic/Subtopic, complete student attempt, XP writes correctly · Confirm no Topic, SubjectMatter, getTopicsByClass in codebase after removal
ADR-003 Spaced Review Widget — Proactive Recall Signal
Context The system tracks quiz performance (pass/fail per topic per student). Currently, that data is only visible backward — in the report card or activity feed. A student who fails a topic receives no signal to revisit it. By the time a tutor notices a pattern, weeks have passed and the student has already forgotten the material. The system has the data to make a proactive recommendation but is not using it.
Decision Add a "Review Due" widget to the student dashboard that surfaces topics where the student has a recent failure (within 7 days), ordered by decay — topics not revisited since failing. Direct link to the relevant quiz. No new collection needed — uses existing quiz_attempts data.
What to build (v1) Hook: useSpacedReview.ts — query quiz_attempts for current student, find topics with status='incorrect' last 7 days, filter out topics already passed since failure, return sorted by decay score (time since failure × attempt count)
UI: SpacedReviewWidget.tsx — compact card: "You missed X on [Topic] — review it" with quiz link · Hidden when 0 items · Badge count on nav
Firestore query: studentId + status + createdAt (no new collection, use existing quiz_attempts)
Feature flag: spacedReview: true in src/config/features.ts
Config: STALE_WINDOW_DAYS default 7 days
What is deliberately excluded Tutor-facing view (v1: student only) · Email/push notification · Tutor-assigned review tasks · SM-2 spaced repetition algorithm
Why this is high-impact The product using its own data to drive behavior. Students who get a prompt to revisit missed material retain it better. The competitive loop (XP + league) keeps students coming back — a review nudge adds genuine learning value on top of the gamification. Without this, Pastpapr is a quiz engine where students practice. With this, it starts to behave like a learning system.
Verification Login as student with failed attempt last 7 days → widget shows topic with quiz link · Login as student with no failures → widget hidden · Click Review → QuizPlayer loads correct quiz · Widget removes item after student passes a retake · npm run test:run passes · Tutor/admin dashboard: widget does not appear

8. Roadmap — Deploying Persona Brainstorm on Any Project

Phase 0 — Preparation (Before the session)
Load project documentation
Clone repo, read .docs/ and .memory/ in full. Aim for 30–60 min reading. Note contradictions, stale states, "in_progress" features with no commits.
Identify 2–3 contradictions
Features marked done but with untested next steps. Deprecated systems still live. Dual code paths with different constants. Version skew between deployed and source.
Prepare Ezekiel's opening question
From the most acute contradiction found. Specific, evidence-based, not exploratory. Structure: "I see X in the code but Y in the history — which is the live state?"
Create ADR.md placeholder
Empty file in project root. Megumi writes completed ADRs here in real time during the session.
Invite Jeremiah's context
Ensure Jeremiah's loaded data covers: product identity, user workflow, known pain points, what's been tried and failed. If Jeremiah can't answer a question from context, flag it as a gap in product knowledge.
Phase 1 — Run the Session (30–45 min)
Orientation (5 min)
Ezekiel opens with known contradiction. One question. Wait for Jeremiah's answer before moving on.
Round 1 — Product loop (5–7 min)
How do users engage with the system daily vs periodically? Where is the engagement hook, and where is the gap? Close surface-level questions fast.
Round 2 — Deep push (7–10 min)
Push on the most real thread from Round 1. Press until Jeremiah gives a concrete pain statement. "Can you give me a specific example?" is the key reframe.
Round 3 — Second theme (5–7 min)
Pivot if a second distinct theme exists. If not, go deeper on Round 2. Aim for 2–3 confirmed improvement themes before writing ADRs.
ADR writing (10 min)
Ezekiel drafts each ADR in real time. Megumi confirms each write to ADR.md. Cap at 3 ADRs total. Each ADR takes ~3 min to draft if the discovery was done right.
Session close — push immediately
ADR.md + CONTEXT.md pushed to project repo before the session ends. No buffering. Skill pushed to harness-engineering-skills.
Phase 2 — Post-Session Actions
Convert ADRs to implementation tasks
Each ADR becomes a planning set: "What to build (v1)" items map to concrete files/features. Break down into test-first tasks. Use TDD on new files.
ADR-002 specific: ESLint sweep
After Curriculum Excision: run grep for Topic, SubjectMatter, getTopicsByClass. Run full test suite. Confirm zero old-model references.
ADR-003 specific: Firestore index check
Spaced Review: verify composite index exists for quiz_attempts (studentId + createdAt + status). If not, deploy index before building the widget.
UX review gate
All three ADRs require tutor/student UX review before implementation starts. Book a walkthrough with the actual user personas before writing code.
Phase 3 — Skill Maintenance
Embed transcript in skill
After each session, add the real transcript excerpt to the skill's "Example" section. Each run improves the skill's precision and examples.
ADR template review
After 3 sessions, review: are the ADRs specific enough that developers can start building without questions? If not, tighten the template.
Pattern logging
If the same class of contradiction surfaces across projects (deprecated code, untested features, dual-write risks), log it as a pattern in .memory/mistakes.json.
Quarterly re-grill
Run a persona brainstorm session at the start of each quarter even if nothing feels broken. Product context shifts — run the session to surface what's changed.

9. Key Rules Cheatsheet

  • One question at a time. Ezekiel asks ONE question, Jeremiah answers, Ezekiel responds. No piling.
  • Jeremiah first. If Judah answers instead of Jeremiah, redirect: "Let's keep Jeremiah in character."
  • No vague ADRs. "Improve UX" is not an ADR. "Add 3-field TutorSessionNoteModal with Firestore write" is. If a developer can't start building from it, it's not ready.
  • ADR comes from friction, not aspiration. The strongest improvements are from real contradictions: code that exists but doesn't work, workflows that create risk, gaps between what the system does and what users need.
  • Cap at 3 ADRs. Three finished ADRs is a strong session. More than three means nothing was committed deeply enough.
  • Document in real time. Megumi writes G-Q# entries to CONTEXT.md as questions close. Writes completed ADRs to ADR.md immediately after drafting.
  • Push before closing. ADR.md + CONTEXT.md pushed to project repo immediately after the session. Skill pushed to harness-engineering-skills. No buffering.
  • Ezekiel never monologues. If Ezekiel is talking for more than 2 sentences, they're not asking a question — they're performing. Stop and ask Jeremiah.

10. When to Run — And When Not To

Run ItDon't Run It
Beginning of a new development cycle (kickoff) When you already have a concrete spec — go straight to planning
Before committing a large investment (new EPIC) When the user wants a list of 10 features — this is a focusing exercise, not a menu
When backlog grooming produces weak priorities When stakeholders are not aligned — Jeremiah must speak with one voice
When the team knows "something is off" but can't name it When the product context hasn't been loaded — Jeremiah answers from memory, not docs
After a major deployment or quarterly review When only 1 person is available (requires developer + client + scribe roles)
